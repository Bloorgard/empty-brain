# Empty Brain — редизайн информационной архитектуры

Дата: 2026-06-20
Статус: согласовано, готово к плану реализации

## Контекст

Empty Brain — Obsidian-хранилище со ссылками на нейросетевые инструменты,
опубликованное как статический сайт через Quartz v4 на GitHub Pages
(`bloorgard.github.io/empty-brain`). Публикуется только папка `content/`.

Хранилище прошло эволюцию: линейный список ссылок → категории по задачам →
постепенно превращается в базу знаний / справочник-учебник. Из-за этого в нём
сейчас сосуществуют несколько несовместимых сущностей в одной плоской папке
`content/AI/` (~60 файлов): страницы-инструменты, страницы-категории, страницы-
компании, теоретические заметки. Навигация держится на ссылках; левая панель
Quartz (Explorer) показывает плоский алфавитный список и фактически бесполезна.

Аудитория: автор (дизайнер, наполняет с компьютера) и гости сайта (ищут
инструмент под тему/задачу). Автор смотрит на область через призму задач дизайна.

## Корневая проблема

Информация по природе **многомерна** (у инструмента сразу несколько измерений:
чем сделан, что на выходе, для кого, как запускается), а иерархия папок —
**одномерна**. Попытка выразить многомерность деревом порождает все наблюдаемые
боли: «куда положить агента — в текст или в автоматизацию», «видео-через-код —
это видео или языковая модель», «оптика дизайнера расходится с категориями».
Дерево не умеет в пересечения; граф (теги, ссылки) — умеет.

## Ключевой принцип: один дом + ниточки

- У каждого инструмента **один дом** в дереве — место, где он физически лежит.
- Пересечения выражаются **тегами и курируемыми подборками** поверх дерева, а
  не дублированием файла.

## Две оси, которые нельзя смешивать

«Модальность» — обманчивое слово, оно склеивает два разных понятия:

- **Чем сделано** — тип модели / технология (Kling — видеомодель). Всегда имеет
  ровно один ответ → годится как ось дерева (дом инструмента).
- **Что на выходе** — задача / результат (получить видео — хоть моделью, хоть
  агентом через HTML). Расходится с технологией в сложных случаях → НЕ ось
  дерева, а отдельный слой поверх.

Поэтому: **дерево строится по технологии**, а **задача живёт отдельным слоем**
курируемых подборок.

## Три слоя контента

| Слой | Жанр доступа | Ось | Глагол |
|------|--------------|-----|--------|
| Справочник | навигатор/каталог (дерево папок, Explorer) | по технологии (чем сделано) | найти |
| Задачи | курируемые подборки (страницы-MOC) | по результату (что на выходе) | решить |
| Учебник | линейный нарратив | от общего к частному | разобраться |

Только **справочник-дерево** физически содержит файлы. Задачи и учебник ничего
не содержат — это слои-указатели поверх, они ссылаются вниз в дерево.

## Главная страница

Не выбирает одну ось как «главную», а предлагает две двери по сценарию прихода:

- **«Разобраться с нуля»** → учебник (линейный маршрут от теории).
- **«Решить задачу»** → список курируемых задачных подборок.

Справочник-дерево НЕ выносится отдельной равноправной дверью (иначе путается с
учебником — оба выглядят как «читать про технологии»). Он понижен до
**постоянного каркаса**: левая панель Explorer + одна корневая страница-каталог,
доступная вторичной ссылкой «смотреть все категории →». Справочник — навигатор,
учебник — нарратив; они разной природы и больше не конкурируют.

## Дерево справочника (по технологии)

```
Языковые модели
   ├ Чат и генерация текста
   ├ Кодинг
   ├ Агенты и автоматизация   (tools · MCP · харнессы)
   └ Запуск локально

Модели изображений
   ├ Генерация (text-to-image)
   ├ Редактирование (image-to-image, inpaint, faceswap, удаление фона…)
   └ Запуск (ComfyUI, A1111…)

Видеомодели
   ├ Генерация
   └ Редактирование

Аудиомодели
   ├ Генерация музыки
   ├ Синтез речи (TTS)
   ├ Транскрибация (STT)
   └ Обработка звука
```

Решения по дереву:

- **Агенты, MCP, харнессы** живут внутри языковых моделей (это надстройка над
  стеком текста), а не отдельным верхнеуровневым доменом.
- **Внутри домена дробим по типу модели** (t2i, i2i, inpaint), а не по задаче.
  Узкоспециализированные модели (faceswap, удаление фона) допустимы как
  подразделы-перечни — это перечень моделей, а не задачный вход.
- **Транскрибация (STT)** — дом в аудио (вход — аудио), стык с текстом
  закрывается тегом, а не дублированием.
- **Компании** перестают быть осью навигации: становятся обычными
  заметками-фактами, доступными через поиск/граф/ссылки.
- **Основы/теория** не входят в дерево справочника — они переезжают в учебник
  как его начало.

## Инфраструктура: доменная vs общая

- **Доменный запуск** (какие оболочки нужны именно для SD / именно для LLM) →
  внутрь соответствующего домена (`… / Запуск`).
- **Общая инфраструктура** (облачные GPU-хостинги RunPod/Vast.ai, платформы-
  агрегаторы) общая для всех моделей → НЕ дробится по доменам (иначе дублируется),
  а живёт отдельным лёгким разделом «Запуск и железо» и/или тегами
  `#облако` / `#локально`.

> Открытый микровопрос реализации: отдельный раздел, тег, или оба. Решается при
> планировании, на архитектуру не влияет.

## Слой задач

Курируемые страницы-подборки по результату («Сделать видео», «Убрать фон»).
Пишутся вручную автором: текст + ссылки `[[…]]` в дерево, с объяснением подходов
(например: «классический путь — модели Kling/Sora; новый путь — через агента и
код»). Это и есть запрошенные читателями «списки под задачу».

- Важные/широкие задачи — ручной курируемый текст с переходами.
- Узкие механические задачи (убрать фон) — допустим авто-список по тегу.
- Гость с задачей входит через явную задачную страницу, а НЕ догадывается кликать
  тег. Тег — вспомогательная арматура, не основной способ навигации.

## Слой учебника

Линейный нарратив, повторяющий технический скелет дерева, но с «нарощенным мясом»:
начинается с теории/основ, далее последовательно ведёт по доменам.

- Учебник **ссылается и пересказывает**, а не копирует контент справочника —
  чтобы не поддерживать одно и то же в двух местах.
- Связь с справочником: ссылки `[[…]]`. Transclusion (`![[Файл#Заголовок]]`)
  применять только для маленьких переиспользуемых блоков (определения), не для
  встраивания крупных разделов.

## Теги-фасеты

Базовые фасеты (модальность, тип) выводятся **из положения в дереве** бесплатно —
файл в `Изображения/Редактирование` и так «изображение + редактирование». Явные
теги нужны только:

- вынесенным-в-отдельный-файл инструментам;
- редким кросс-пересечениям (`#через-код`, `#для-дизайна`, `#агенты`, `#облако`).

Набор фасетов держим **коротким и закрытым**, чтобы не расплылся в сотню
стихийных тегов. Финальный список фасетов фиксируется на этапе плана.

## Гранулярность файлов: органический порог

Глубина не фиксируется заранее. Правило рекурсивное:

> Секция в файле разрослась → выноси в отдельный файл.
> Группа файлов разрослась → выноси в папку.

- Инструмент по умолчанию = **строка в списке** подраздела (ссылка + краткий
  комментарий). Это даёт сканируемый обзор «что накопилось».
- Отдельный файл заводится, только когда инструмент перерос строку (туториал,
  глубокий разбор, свои под-ссылки) — тогда из списка ведёт `[[ссылка]]`.
  Так уже сложилось с `ComfyUI`, `Stable Diffusion`, `suno tutorial`.
- Перемещение безопасно: Obsidian/Quartz резолвят `[[ссылки]]` по имени файла,
  а не по пути.

Будущая атомизация (файл на инструмент с frontmatter) оправдана только если
дойдёт до отдельного интерфейса (например, Telegram-бот, отдающий инструмент по
запросу). Сейчас — YAGNI, файлы ради гипотезы не плодим. При этом структура
(положение в дереве) уже служит machine-readable слоем для такого интерфейса.

## Сопутствующие артефакты

- **`CLAUDE.md` в корне репозитория** — короткая операционная памятка для агентов
  (как раскладывать новый инструмент по правилам выше). Создаётся как выжимка из
  этого спека, отдельным шагом. Claude Code подхватывает его автоматически.
- Этот спек лежит в `docs/`, вне `content/`, поэтому на сайт не публикуется.

## За рамками этого редизайна (не делаем сейчас)

- Telegram-бот и атомизация файлов под него.
- Массовое переписывание содержимого страниц (редизайн — про структуру, не про
  ревизию каждого текста).
- Кастомизация темы/верстки Quartz сверх того, что нужно для двух дверей и
  корректного Explorer.
